Data Structures: Stack, Queue, Heap, AVL Tree, HashMap

Five structures, each solving a problem the others cannot. Explained from scratch with diagrams. Pure concept here, no project context yet. That comes in Part 4.

S
Stack
LIFO — Last In, First Out

Imagine a stack of plates in a kitchen. You put a new plate on top. When you need one, you take from the top. You cannot take from the middle without removing everything above it first.

That is the entire idea. The last thing you put in is the first thing you get out. This rule is called LIFO: Last In First Out.

Two operations

  • push — add something to the top
  • pop — remove and return the top item
  • peek — look at the top item without removing it

Stack grows upward. push adds to the top. pop removes from the top. The bottom item is always the oldest.

Implementation with a linked list

The top of the stack is the HEAD of the linked list. Push means insert at HEAD. Pop means remove from HEAD. Both are instant, single step operations regardless of how many items the stack holds.

Real world parallel: Your browser's back button is a stack. Every page you visit gets pushed onto it. When you press back, the current page is popped and you return to the one below it.
Q
Queue
FIFO — First In, First Out

The opposite rule from a stack. Think of people standing in line at a hospital counter. The first person to arrive gets served first. No jumping the line, no exceptions.

This is called FIFO: First In First Out. Order of arrival is the only thing that matters.

Two operations

  • enqueue — add to the back of the line
  • dequeue — remove from the front of the line

Queue: new items join at the rear, items leave from the front. Arrival order is always preserved.

Implementation with a linked list

A queue holds two pointers: front and rear. Enqueue adds a new node at rear and updates the rear pointer forward. Dequeue removes the node at front and moves the front pointer to the next node.

Stack vs Queue: A stack has one access point at the top. A queue has two: you add at one end and remove from the other. The order of insertion is respected perfectly, which is exactly what you want when processing cases in the sequence they were reported.
H
Heap
Priority-based — highest priority always on top

A heap solves a problem neither a stack nor a queue can solve: what if you do not care about arrival order at all, but only about which item is the most critical right now?

Think of a hospital emergency room. A person with a headache arrived before someone with a heart attack. The heart attack patient should not wait simply because they arrived second. Urgency overrides order of arrival entirely.

A max heap always keeps the highest priority item at the very top. Whenever you ask for the next item, you always get the most critical one available, instantly.

The one rule: parent is always greater than its children

A heap is shaped like a tree. Every parent node has a value greater than or equal to both its children. This single rule guarantees the maximum value sits at the root, always, no matter how many items are in the heap.

Max heap. The root always holds the largest value. Every parent is greater than its children. 95 is always instantly accessible at the top.

Implementation in SENTINEL

We maintain a sorted linked list in descending order of priority score. The head is always the highest priority item. Inserting a new item means walking to find the correct position and splicing in. This preserves the heap property using pure linked list logic with no array involved.

The core guarantee: No matter how many items the heap holds, the most critical one is always at the top. You never search for it. One step access, always.
AVL
AVL Tree
Self-balancing Binary Search Tree

To understand an AVL tree you need to understand a regular binary search tree first, and then understand why it can fail.

Binary Search Tree

A BST follows one rule at every node: values smaller than the current node go left, values larger go right. Searching is fast because at every step you eliminate half the remaining possibilities.

BST rule at every node: go left if smaller, go right if larger. Finding 60 takes only 3 steps: check 50 (go right), check 70 (go left), found 60.

The problem: imbalance

Insert values in sorted order, say 10, 20, 30, 40, and every node goes right because each is larger than the last. The tree becomes a diagonal line. All speed advantage is gone. Search now takes as long as checking every item one by one.

Inserting sorted values breaks the BST entirely. Every node only has a right child. The tree is now a slow diagonal line.

The AVL fix: automatic rotation

After every insertion, an AVL tree checks whether either subtree has grown too tall compared to the other. This difference in height is called the balance factor. If it exceeds 1, the tree performs a rotation: a local restructuring where a child takes the place of its parent. The BST ordering is preserved. The tree is balanced again. Speed is restored.

AVL rotation. 20 takes the root. 10 and 30 are now equal-depth children. Balance restored, speed restored.

The guarantee: An AVL tree stays balanced no matter what order data arrives in. Search, insert, and delete remain fast even with thousands of records. This is why it is used for range queries in SENTINEL rather than scanning the queue linearly.
HM
HashMap
Key-Value Store — near-instant lookup

A HashMap solves one problem very well: how do you find a specific record instantly without searching through everything?

Think of a physical dictionary. You do not read from page one to find a word starting with J. You flip directly to the J section. A HashMap does something mathematically equivalent for any kind of key.

How the hash function works

You give it a key, like a patient ID. It runs the key through a hash function that converts it into a number. That number is used as a direct index into a table of slots. Your data goes into that exact slot. When you want it back, the same key produces the same number and you go directly there.

P003 goes into the hash function, comes out as index 7, system goes directly to slot 7. No scanning through slots 0 to 6.

Collisions and chaining

Sometimes two different keys produce the same index. This is a collision. The solution is chaining: each slot holds a small linked list. If two keys land in the same slot, both nodes live in that slot's chain. You walk the short chain to find your specific key.

Why the speed difference is enormous: With 50,000 records, a linear search checks up to 50,000 items. A HashMap goes directly to one slot and walks a chain of maybe two or three. The hash function converts an O(n) problem into something approaching O(1).
