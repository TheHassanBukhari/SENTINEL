Data Structures: Stack, Queue, Heap, AVL Tree, HashMap

1. Stack

LIFO: Last In, First Out

Imagine a stack of plates in a kitchen. You put a new plate on top. When you need a plate you take from the top. You cannot take from the middle or the bottom without removing everything above it first.

That is the entire idea of a stack. The last thing you put in is the first thing you get out. This rule is called LIFO: Last In First Out.

Two operations only

  • push: add something to the top
  • pop: remove and return the top item

There is also peek which lets you look at the top item without removing it.

Stack grows upward. push adds to the top. pop removes from the top. The bottom item is always the oldest.

How it is implemented with a linked list

The top of the stack is simply the HEAD of the linked list. Push means insert at HEAD. Pop means remove from HEAD. Both are instant, one step operations regardless of how many items are in the stack.

Real world use: Your browser's back button is a stack. Every page you visit gets pushed. When you press back, the current page gets popped and you return to the one below it.

2. Queue

FIFO: First In, First Out

A queue is the opposite rule. Think of people standing in line at a hospital registration counter. The first person to arrive gets served first. No jumping the line.

This rule is called FIFO: First In First Out.

Two operations

  • enqueue: add to the back of the line
  • dequeue: remove from the front of the line

Queue: new items join at the rear, items leave from the front. Order of arrival is always respected.

How it is implemented with a linked list

A queue keeps two pointers: front and rear. Enqueue adds a new node at the rear and updates the rear pointer. Dequeue removes the node at front and moves the front pointer forward.

Key difference from stack: A stack has one access point (the top). A queue has two: you add at one end and remove at the other. The order of insertion is preserved perfectly.

3. Heap

Priority-based: highest priority always on top

A heap solves a problem neither a stack nor a queue can solve: what if you do not care about arrival order, but only about which item is most important right now?

Imagine the hospital emergency room again. A person with a headache walked in before someone with a heart attack. The heart attack patient should not wait simply because they arrived second. Urgency overrides order.

A max heap always keeps the highest priority item at the top. Whenever you ask for the next item, you always get the most critical one, not the oldest one.

The rule: parent is always greater than its children

A heap is shaped like a tree. Every parent node has a value greater than or equal to both its children. This single rule guarantees the maximum value is always at the root, the very top.

Max heap. The root always holds the largest value. Every parent is greater than its children. 95 is always available instantly at the top.

How it is implemented with a linked list in SENTINEL

We maintain a sorted linked list where nodes are kept in descending order of priority score. The head of the list is always the highest priority item. Inserting a new item means walking the list to find the correct position and splicing it in. This preserves the heap guarantee using pure linked list logic.

The core guarantee: No matter how many items are in the heap, the most critical item is always instantly accessible at the top. You never have to search for it.

4. AVL Tree

Self-balancing Binary Search Tree

To understand an AVL tree, you need to understand a regular binary search tree first.

Binary Search Tree (BST)

A BST is a tree where every node follows one rule: values smaller than the current node go to the left, values larger go to the right. This means searching is fast because at every step you eliminate half the remaining nodes.

BST. At every node: go left if smaller, go right if larger. Searching for 60 takes only 3 steps: check 50 (go right), check 70 (go left), found 60.

The problem: imbalance

If you insert data in sorted order, say 10, 20, 30, 40, 50, every node is larger than the previous so every node goes to the right. The tree becomes a straight line. You have lost all speed advantage. Searching now takes as long as scanning every item one by one.

Inserting sorted values breaks the BST. Every node only has a right child. The tree is now just a slow diagonal line.

AVL fix: automatic rotation

An AVL tree checks itself after every insertion. It measures the height difference between the left and right subtrees of every node. This difference is called the balance factor. If the balance factor goes beyond 1 in either direction, the tree rotates itself to rebalance.

A rotation is simply a local restructuring where a child takes the place of its parent, keeping all the BST ordering rules intact. After rotation the tree is balanced again and search speed is restored.

AVL rotation. The unbalanced chain on the left is rotated so 20 becomes the root. Now both sides are equal and search is fast again.

The guarantee: An AVL tree is always balanced no matter what order you insert data. Search, insert, and delete are all fast even with thousands of records.

5. HashMap

Key-Value Store: near instant lookup

A HashMap solves one problem: how do you store and retrieve data instantly without searching through everything?

Think of a dictionary. You do not read every page to find a word. You know that words starting with J are near the middle. You flip directly there. A HashMap does something similar but mathematically.

How it works: the hash function

You give it a key, like a patient ID. It runs that key through a mathematical function called a hash function. The hash function converts the key into a number. That number is used as an index into an array of slots. Your data goes into that slot.

When you want the data back, you give the same key, the same hash function produces the same number, and you go directly to that slot. No searching.

Key P003 goes into the hash function, comes out as index 7, and the system goes directly to slot 7. No searching through slots 0 to 6.

Collisions and chaining

Sometimes two different keys produce the same index. This is called a collision. The solution is chaining: each slot holds a small linked list. If two keys land in the same slot, both their nodes sit in that slot's linked list. You walk the short chain to find your specific key.

Why this matters: With 50,000 patient records, a normal search checks up to 50,000 records. A HashMap checks one slot plus a tiny chain. The difference in speed is enormous.
