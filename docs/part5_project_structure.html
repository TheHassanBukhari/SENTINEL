Header Files vs Source Files

In C++ you almost always see files in pairs: a .h file and a .cpp file. Understanding why that split exists matters before reading any C++ project.

The .h file is a declaration

A header file declares what exists. It says: there is a class called Stack, it has a method called push, push takes these parameters and returns this type. It is a contract. It does not contain how push works, only that it exists and what calling it looks like.

The .cpp file is the definition

A source file defines what was declared in the header. It contains the actual implementation. The real logic. How push works step by step.

Why this separation exists

When file A needs to use something from file B, file A includes B's header. That gives A everything it needs to compile. A does not need to see B's implementation. It just needs to know the shape of what it is calling. This keeps files focused and lets large projects compile efficiently.

SENTINEL uses header-only design. All logic lives inside .h files with no separate .cpp files. This is a valid approach for projects of this size. It trades some compile efficiency for simplicity, and since everything compiles from a single main.cpp include chain, it works cleanly.

The include guard

Every header file starts with #pragma once. This tells the compiler: if you have already seen this file, skip it. Without it, including the same header from two places would define the same class twice and cause a compile error.

Project File Structure

Two CSVs in data/, one entry point, ten header files. Each file owns one responsibility.

What Each File Does

Stack.h
Per-patient investigation log
StackNode holds timestamp, day number, event string, officer note, and a next pointer.
Stack implements push, pop, peek, isEmpty, getSize, and display.
No dependencies. Includes only string and iostream. The bottom of the entire dependency chain.
Case.h
The central data object of the system
Case holds patient ID, name, age, area code, day registered, seven symptom booleans, status string, risk score integer, a Stack for the investigation log, and a next pointer for queue chaining.
calculateRiskScore() lives here and runs automatically in the constructor.
Depends on: Stack.h
Queue.h
Master case pipeline
Queue holds front and rear pointers and a size counter.
Methods: enqueue, dequeue, getFront, isEmpty, findById, displayAll, displayByDay, displayByArea.
displayByDay and displayByArea traverse the queue and filter inline, producing snapshots without a separate data structure.
Depends on: Case.h
Heap.h
Risk priority engine
HeapNode holds patient ID, risk score, and next pointer.
Heap maintains a linked list in descending order of risk score. Head is always the highest risk patient.
Methods: insert, extractMax, peekMax, isEmpty, display, updateScore.
updateScore removes the old entry and reinserts at the correct position, keeping the sort intact.
Depends on: Case.h
AVLTree.h
Date-indexed self-balancing search tree
AVLNode holds patient ID, day integer, height, left pointer, right pointer.
AVLTree implements rotateLeft, rotateRight, balance factor calculation, and the balance dispatcher.
Public methods: insert, searchRange, displayAll.
searchRange(d1, d2) uses inorder traversal with branch pruning: subtrees entirely outside the range are skipped without visiting.
No dependencies beyond string and iostream.
HashMap.h
Dual lookup: person profiles and location index
PersonEntry holds key string, Case pointer, and next pointer for collision chaining.
LocationEntry holds area code string, array of patient IDs, count, and next pointer.
HashMap contains personTable and locationTable, each 101 slots wide.
Hash function: polynomial rolling hash multiplying by 31 per character, then modulo 101.
Methods: insertPerson, getPerson, insertLocation, displayLocation, displayAllLocations.
Depends on: Case.h
Graph.h
Weighted contact network: foundation of all analysis
AdjNode holds neighbor ID, risk weight, contact day, location type, and next pointer.
GraphNode holds patient ID and head of its adjacency list.
Graph contains an array of GraphNodes plus a parallel ID string array. getIndex maps ID to index via linear search.
bfsWaveSimulation uses an internal integer-indexed queue to perform BFS and prints each wave level.
No dependencies beyond string and iostream.
Dijkstra.h
Shortest transmission path tracer
One public method: findPath(graph, srcId, destId).
Uses integer arrays for dist, visited, and parent. No dynamic allocation inside the algorithm.
Edge weight inversion: distance added is (10 minus risk weight), so high-risk contacts appear as short distances. Dijkstra then finds the highest-probability chain naturally.
Path reconstructed by following parent array backwards from destination, then printed in reverse.
Depends on: Graph.h
Prims.h
Minimum containment network
One public method: findMinContainmentNetwork(graph).
Integer arrays for inMST flags, key values, and parent IDs.
At each step finds the unvisited node with the smallest key, adds it to the MST, then relaxes its edges.
Prints each selected contact link as a quarantine target, with its risk weight, plus the total network risk score.
Depends on: Graph.h
DataLoader.h
CSV reader: populates all structures simultaneously
Two methods: loadPatients and loadContacts.
loadPatients opens patients.csv, skips the header, reads each row, creates a Case, pushes an initial stack event, then inserts into Queue, Heap, AVL Tree, both HashMaps, and Graph in sequence.
loadContacts opens contacts.csv and calls addEdge for each row.
Depends on: Queue.h, Heap.h, AVLTree.h, HashMap.h, Graph.h
main.cpp
Entry point, global state, menu loop
Declares all seven structures as global objects accessible from every function.
On startup: prints the SENTINEL header, calls DataLoader to populate from CSV.
Runs a do-while loop with a 16-option menu. A switch statement routes each choice to the correct method on the correct structure.
Depends on: all header files.

How the Files Connect

The dependency chain flows top to bottom. Stack.h is at the bottom because nothing depends on it except Case.h. main.cpp is at the top because it depends on everything.

How Data Flows: CSV to All Structures

When the program starts, main calls DataLoader. Here is the exact sequence for each line in patients.csv:

1
CSV line read as string. Split by comma into 13 fields: ID, name, age, area, day, seven symptom booleans, status.
2
New Case object created with new Case(...). All fields set. calculateRiskScore() runs automatically in the constructor.
3
Initial event pushed to this case's personal Stack: day of registration, event description, intake note.
4
caseQueue.enqueue(c): case joins the master queue at the rear in arrival order.
5
riskHeap.insert(id, score): patient ID and risk score enter the heap, sorted to correct position.
6
patientTree.insert(id, day): patient enters the AVL tree indexed by day. Tree rebalances if needed.
7
patientMap.insertPerson(id, c): patient ID maps to case pointer in the person hashmap.
8
patientMap.insertLocation(area, id): patient ID added to their area code's list in the location hashmap.
9
contactGraph.addNode(id): a graph node is created for this patient.

After all patients load, contacts.csv runs. Each line calls addEdge with two patient IDs and a risk weight, building all connections in the graph.

One patient, nine operations, all seven structures updated. After DataLoader finishes, the system is fully populated and every menu option works immediately.

Compiling and Running

Open a terminal, navigate to the project folder, and run:

cd project
g++ main.cpp -o sentinel
./sentinel

The compiler reads main.cpp. Because main.cpp includes all header files, the compiler sees the entire codebase in one pass. The output binary is named sentinel.

CSV path error: If the program says it cannot find patients.csv, make sure you are running from inside the project/ folder. The paths are relative: data/patients.csv means the program looks for a data subfolder wherever it was launched from.

Design Decisions

Every structural choice in SENTINEL has a specific reason. These are the questions most likely to come up when explaining the project.

QuestionReasoning
Why linked lists instead of arrays? The number of patients in an outbreak is unpredictable. Linked lists grow without a predefined limit and allocate memory only for what exists. No wasted reserved slots.
Why does each patient have their own stack? Each patient has a unique sequence of events. A global stack would mix events from all patients. Embedding a stack inside each case keeps the history self-contained and directly tied to the patient it belongs to.
Stack vs Queue: what is the difference in this system? Stack is LIFO, per patient, for investigation history. Queue is FIFO, global, for case processing order. They solve opposite ordering problems for different purposes.
How does Dijkstra find a transmission path? Edge weights are inverted: distance added is (10 minus risk weight). High-risk contacts become short distances. Dijkstra finds shortest distance, which equals the highest-probability chain.
What separates Dijkstra from Prim's? Dijkstra finds the best path between two specific patients. Prim's finds the minimum set of connections holding an entire cluster together. Different questions, different outputs.
Why AVL Tree when there is already a HashMap? HashMap gives instant lookup by ID but cannot do range queries or maintain sorted order. AVL Tree enables range search by day and sorted timeline display. Two different problems.
Why two hashmaps? One maps patient ID to profile for fast individual lookup. One maps area code to patient list for geographic analysis. Different keys, different purposes, both needed.
What does BFS actually simulate? Starting from any patient, BFS shows who gets exposed wave by wave: day 1 contacts, then their contacts on day 2, and so on. It models how infection spreads outward through the contact network.
Does the system save data after changes? The system loads from CSV at startup and operates in memory. Changes are not written back. This is a known limitation for the current version.

Every structure in SENTINEL exists because the problem required it. The epidemic domain maps naturally onto graphs, priority queues, and search trees. The system is not a showcase of data structures. The data structures are a consequence of modeling the problem honestly.
