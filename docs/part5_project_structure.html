1. Header Files vs Source Files

In C++ projects you almost always see pairs of files: a .h file and a .cpp file. Understanding why they exist as a pair is important before reading the project.

The .h file: the declaration

A header file declares what exists. It says: there is a class called Stack, it has a method called push, push takes these parameters and returns this type. It is a promise. A contract. It does not contain the actual logic of how push works, only that push exists and what its signature looks like.

The .cpp file: the definition

A source file defines what was declared. It contains the actual implementation. The real code. How push works step by step.

Why this separation exists

When file A needs to use something from file B, file A includes B's header. This gives A the declarations it needs to compile. A does not need to see B's full implementation to use it. It just needs to know the shape of what it is calling.

This means large projects can compile faster, files stay focused, and changes to an implementation do not force everything that uses it to recompile from scratch.

In SENTINEL we kept all logic inside header files only. No separate .cpp files. This is called header only design and it is a valid approach especially for smaller projects and template heavy code. It trades some compilation efficiency for simplicity. For a project of this size it is perfectly fine.

The include guard

Every header file starts with #pragma once. This tells the compiler: if you have already included this file once, do not include it again. Without this, if two files both include Stack.h, the Stack definition would appear twice and cause errors.

2. Project File Structure

Complete project layout. Two CSV files in data folder. One main entry point. Ten header files each responsible for one thing.

3. What Each File Does

Stack.h
Defines the StackNode struct and the Stack class
StackNode holds timestamp, day number, event string, officer note, and a next pointer.
Stack has push, pop, peek, isEmpty, getSize, and display methods.
This file has no dependencies. It includes nothing except string and iostream. It is the foundation everything else builds on.
Case.h
Defines the Case struct: the central data object of the entire system
Case holds patient ID, name, age, area code, day registered, all seven symptom booleans, status string, risk score integer, a Stack object for investigation log, and a next pointer for queue chaining.
The calculateRiskScore() method lives here. It reads the seven booleans and returns the total score.
Depends on: Stack.h
Queue.h
Defines the master case pipeline
Queue holds front and rear pointers and a size counter.
Methods: enqueue, dequeue, getFront, isEmpty, getSize, findById, displayAll, displayByDay, displayByArea.
displayByDay traverses the queue and prints only cases matching a given day integer.
displayByArea traverses and filters by area code string.
Depends on: Case.h
Heap.h
Defines the risk priority engine as a sorted linked list
HeapNode holds patient ID string, risk score integer, and next pointer.
Heap maintains a linked list sorted in descending order of risk score. Head is always highest risk.
Methods: insert, extractMax, peekMax, isEmpty, getSize, display, updateScore.
updateScore removes the old entry and reinserts with the new score, maintaining sort order.
Depends on: Case.h
AVLTree.h
Self-balancing binary search tree indexed by outbreak day
AVLNode holds patient ID, day integer, height integer, left pointer, right pointer.
AVLTree contains all rotation logic (rotateLeft, rotateRight), balance factor calculation, and the balance method that decides which rotation to apply.
Public methods: insert, searchRange, displayAll.
searchRange(d1, d2) performs an inorder traversal that prunes branches outside the range. Fast and precise.
No dependencies beyond string and iostream.
HashMap.h
Dual hashmap: person profiles and location index
PersonEntry holds a key string, a Case pointer, and a next pointer for chaining.
LocationEntry holds an area code string, an array of patient ID strings, a count, and a next pointer.
HashMap contains two separate tables: personTable and locationTable, each of size 101 slots.
The hash function uses a polynomial rolling hash: multiplies running total by 31 and adds each character's ASCII value, then takes modulo 101.
Methods: insertPerson, getPerson, insertLocation, displayLocation, displayAllLocations.
Depends on: Case.h
Graph.h
Contact network: the skeleton of the entire system
AdjNode holds neighbor patient ID, risk weight, day of contact, location type string, and next pointer.
GraphNode holds the patient ID and the head of its adjacency linked list.
Graph contains an array of GraphNode objects and a parallel array of node ID strings. getIndex performs linear search to map ID to array index.
Methods: addNode, addEdge, displayGraph, getNeighbors, getWeight, getNodeCount, getNodeId, nodeExists, bfsWaveSimulation.
bfsWaveSimulation uses an internal array based queue with a front and rear integer to perform BFS and prints each wave level.
No dependencies beyond string and iostream.
Dijkstra.h
Shortest transmission path between two patients
Dijkstra class with one public method: findPath(graph, srcId, destId).
Uses integer arrays for dist, visited, and parent tracking. No dynamic allocation inside the algorithm itself.
Edge weight is inverted: distance added is (10 minus risk weight) so high risk contacts become short distances and Dijkstra naturally finds the highest risk path.
Path is reconstructed by following the parent array backwards from destination to source, then printed in reverse.
Depends on: Graph.h
Prims.h
Minimum containment network across all connected cases
Prims class with one public method: findMinContainmentNetwork(graph).
Uses integer arrays for inMST flags, key values, and parent tracking.
At each iteration finds the node with the smallest key value not yet in the MST, adds it, then relaxes all its edges.
Prints each selected edge as a contact link to break, with its risk weight, and the total network risk score.
Depends on: Graph.h
DataLoader.h
Reads CSV files and populates all structures simultaneously
DataLoader class with two methods: loadPatients and loadContacts.
loadPatients opens patients.csv, skips the header line, reads each row, splits by comma, creates a Case object, pushes an initial stack event, then inserts into Queue, Heap, AVL Tree, person HashMap, location HashMap, and Graph.
loadContacts opens contacts.csv, reads each row, and calls addEdge on the Graph with the two patient IDs and risk weight.
Depends on: Queue.h, Heap.h, AVLTree.h, HashMap.h, Graph.h
main.cpp
Entry point, global state, menu system
Declares all seven data structures as global objects so every function can access them without passing parameters everywhere.
On startup: prints the SENTINEL header, calls DataLoader to populate all structures from CSV.
Runs a do-while loop displaying a 16 option menu. Each option calls the relevant method on the relevant structure.
Handles new patient registration, status updates, event logging, all display operations, and all algorithmic operations.
Depends on: All header files.

5. How Data Flows: From CSV to All Structures

When the program starts, main calls DataLoader. Here is the exact sequence of what happens for each line in patients.csv:

1
CSV line is read as a string. It is split by comma into 13 fields: ID, name, age, area, day, seven boolean columns, status.
2
A new Case object is created on the heap using new Case(...). All fields are set. calculateRiskScore() runs automatically in the constructor.
3
An initial investigation event is pushed onto this case's personal Stack: day of registration, event description, initial note.
4
caseQueue.enqueue(c): the case pointer enters the master queue at the rear.
5
riskHeap.insert(id, score): patient ID and risk score enter the heap, sorted into correct position.
6
patientTree.insert(id, day): patient ID and day enter the AVL tree. Tree rebalances if needed.
7
patientMap.insertPerson(id, c): patient ID maps to case pointer in person hashmap.
8
patientMap.insertLocation(area, id): patient ID is added to their area code's list in location hashmap.
9
contactGraph.addNode(id): a graph node is created for this patient.

After all patients are loaded, contacts.csv is processed. Each line adds one weighted edge between two existing graph nodes.

One patient, nine operations, seven structures all updated simultaneously. After DataLoader finishes, the entire system is populated and every menu option is ready to use.

7. Compiling and Running

On Zorin OS open a terminal, navigate to the project folder, and run:

cd project
g++ main.cpp -o sentinel
./sentinel

The compiler reads main.cpp. Because main.cpp includes all the header files, the compiler sees all the code it needs in one pass. The output binary is named sentinel. Running it starts the program.

If you get an error saying patients.csv not found, make sure you are running the program from inside the project folder, not from somewhere else. The CSV paths are relative: data/patients.csv means the program looks for a data folder inside wherever it is run from.

8. Viva Preparation: Questions You Will Likely Be Asked

QuestionAnswer to give
Why did you use a linked list instead of arrays?Dynamic size. We do not know how many patients will be recorded. Linked lists grow without limits and without wasted reserved space.
What is the difference between your Stack and Queue?Stack is LIFO, per patient, for investigation history. Queue is FIFO, global, for case processing order. Opposite access patterns for different purposes.
Why does each patient have their own stack?Because each case has its own timeline of events. A global stack would mix events from all patients. This way the case file is self contained.
How does Dijkstra work in your system?We invert edge weights so high risk contacts become short distances. Dijkstra finds the shortest distance path which in our case is the highest probability transmission chain.
What is the difference between Dijkstra and Prim's?Dijkstra finds the best path between two specific patients. Prim's finds the minimum set of edges connecting all patients in a cluster. Different goals entirely.
Why AVL Tree and not just HashMap for patient lookup?HashMap gives instant lookup by ID but cannot do range queries or maintain sorted order. AVL Tree gives us range search by day and chronological display. They solve different problems.
Why two hashmaps?One maps patient ID to their profile for fast individual lookup. One maps area code to list of patients for fast geographic analysis. Different keys, different purposes.
What does BFS do in SENTINEL?Simulates the outbreak spreading wave by wave from a starting patient. Shows who gets exposed in day 1, day 2, day 3 of the spread from that source.
What is the risk score based on?Seven binary factors: fever, cough, breathing difficulty, large gathering attendance, high contact job, known contact with confirmed case, age above 60 or comorbidity. Maximum 130 points.
How does your system save data?Currently the system loads from CSV at startup and operates in memory. Data is not persisted back to CSV after changes. This is a known limitation.
Final note: You built a system that models a real epidemic. Every structure has a reason. Every algorithm solves a problem that would be unsolvable without it. When your examiner asks why you chose something, the answer is never because we needed to use it. The answer is always because the problem required it.
