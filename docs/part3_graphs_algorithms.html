Graphs, BFS, Dijkstra, and Prim's Algorithm

The three most algorithmically demanding parts of SENTINEL all operate on graphs. Understanding the graph structure first makes everything else click into place.

G
What Is a Graph
Nodes connected by edges

A graph is the most general way to represent relationships between things. Two components, nothing more:

  • Nodes (also called vertices): the things. A person, a city, a router, a patient.
  • Edges: the connections between them. A contact, a road, a cable, a transmission event.

That is all a graph is. Dots and lines. But this simple idea can model almost any real world system involving relationships.

A weighted graph. Five nodes, six edges, each edge carries a number representing cost, distance, or risk.

Weighted vs unweighted

An unweighted graph simply records whether two nodes are connected. A weighted graph assigns a number to each edge. That number can represent anything meaningful: distance, cost, duration, transmission risk.

Directed vs undirected

An undirected graph means connections go both ways. If A connects to B then B connects to A. A directed graph has one-way connections, like a one-way street. SENTINEL uses undirected weighted edges because contact between two people goes both ways.

How a graph is stored in code

Each node is an object. Each node holds a linked list of its neighbors, called an adjacency list. Each entry in that list contains the neighbor's ID and the edge weight. The whole graph is a collection of these node objects connected by their lists.

Every real world system involving relationships can be modeled as a graph. Social networks, road maps, the internet, power grids, disease spread. The three algorithms below are how you extract meaning and make decisions from that graph.
BFS
Breadth First Search
Explores level by level outward from a starting node

BFS answers: starting from one node, in what order do you reach all other nodes if you always explore the closest ones first?

Imagine dropping a stone in still water. Ripples spread outward in rings. The first ring is closest. The second ring is one step further. The third is two steps out. BFS explores a graph exactly like this, ring by ring, level by level, never jumping ahead.

How it works

1
Start at the source node. Mark it visited. Add it to the queue.
2
Dequeue the front node. Find all its unvisited neighbors. Mark each visited and add each to the back of the queue.
3
Repeat. Dequeue the next node. Visit its unvisited neighbors. Add them to the queue.
4
Continue until the queue is empty. Every reachable node has been visited in wave order.

BFS from A. Visits A first, then B and C one step away, then D E F G two steps away. Always completes one full wave before moving to the next.

Why BFS needs a Queue specifically

The Queue enforces the wave order. Neighbors of A go to the back: B and C are queued. Processing B adds D and E to the back. But C is still at the front, ahead of D and E, so C is processed before them. FIFO is exactly what produces the ring-by-ring exploration. Without the queue's ordering, BFS collapses.

The Queue is not just a helper here. It is the mechanism that makes BFS work. The wave pattern emerges directly from FIFO ordering. Change it to a stack and you get a completely different algorithm (DFS) that dives deep rather than spreading wide.
D
Dijkstra's Algorithm
Shortest weighted path between two nodes

BFS finds the path with the fewest hops. Dijkstra finds the path with the lowest total weight. These are different problems with different answers.

A road map example: city A to city D. Route 1 uses two roads totaling 900 km. Route 2 uses four roads totaling 320 km. BFS picks Route 1 because it has fewer hops. Dijkstra picks Route 2 because it has lower total cost. For epidemic tracing, the lowest cost path is the highest probability infection chain.

The core idea

Every node starts with distance infinity except the source which starts at zero. Repeatedly pick the unvisited node with the lowest current distance. Visit it. For each of its neighbors, check whether going through this node produces a shorter total distance than what the neighbor currently has. If yes, update it. Repeat until the destination is reached.

Worked example

Dijkstra finds A to D costs 5 via C, beating the 9-cost routes through B and direct. It always finds the mathematically guaranteed minimum.

The guarantee: Dijkstra never makes a distance worse. It only improves. When a node is marked visited, its distance is final. It cannot later be found to be wrong. This is what makes the output trustworthy, not just a good guess.
MST
Prim's Algorithm
Builds the minimum spanning tree of a graph

Dijkstra answers: cheapest path from A to B. Prim's answers something entirely different: what is the minimum set of edges that keeps every single node connected to every other node?

This result is called a Minimum Spanning Tree. A spanning tree connects all nodes with no cycles using the minimum total edge weight possible.

Imagine 8 villages needing electrical cable. You do not need every village directly connected to every other. You just need every village reachable through some path. And you want the least total cable used. Prim's finds exactly which connections achieve that.

How it works

1
Start at any node. Add it to the MST set.
2
Look at every edge connecting a node inside the MST to a node outside it. Pick the one with the lowest weight.
3
Add that edge and its new node to the MST set.
4
Repeat until all nodes are included. The result is the minimum spanning tree.

Prim's picks 3 edges from 6. All 4 nodes remain connected. Total cost 12 is the minimum achievable. Three edges were skipped entirely.

The confusion point: Both algorithms look similar in code. Both pick the minimum at each step. The difference is what they minimize. Dijkstra minimizes distance from the source to each node. Prim's minimizes the weight of the edge being added to the growing tree. Different goals, completely different outputs.
